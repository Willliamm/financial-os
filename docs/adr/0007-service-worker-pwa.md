# 0007 — Hand-rolled service worker for PWA

## Status

Accepted

## Context

The app is offline-first (see ADR 0001). To truly work offline and to be
installable, it should be a **PWA** (Progressive Web App) — a web app that can
be installed and can load with no network.

A PWA needs a **service worker**: a script the browser runs in the background
that can cache files and serve them offline.

There are tooling options, such as `next-pwa` or Workbox, that generate a
service worker during the build. But:

- The app is a **static export** with a custom build path (see ADR 0001), and
  these tools add build complexity and config we do not need.
- We want **full control** over what is cached and when, and a worker that is
  simple to read and audit for an offline finance app.
- We must avoid caching getting in the way during development.

## Decision

Ship a **hand-rolled service worker** at **`public/sw.js`**.

- The file is written and maintained by us, not generated by a plugin.
- It is **registered in production only**. In development the service worker is
  not registered, so it never serves stale cached files while coding.
- It handles the offline caching the PWA needs, with logic we control directly.

## Consequences

**Good**

- Full, readable control over caching for a privacy-sensitive offline app.
- No extra PWA build plugin or its configuration to maintain.
- Prod-only registration keeps development free of stale-cache surprises.

**Bad / trade-offs**

- We own the service worker logic, including cache-versioning and update
  behavior. Mistakes here can serve stale assets to users, so changes need
  care.
- Service worker behavior differs between dev (off) and prod (on), so
  offline behavior must be tested against a production build (`npm run build`,
  then serve `out/`).
- No automatic precache manifest; cached assets are managed by hand.
